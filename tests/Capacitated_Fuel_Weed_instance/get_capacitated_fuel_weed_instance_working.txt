So why?

The initial solution strategy has to create a feasible solution. Thus even if a feasible solution does exist, if the solver can't create one using its initial search, it just gives up.

A comment from the or-tools forum: (https://groups.google.com/forum/#!searchin/or-tools-discuss/infeasible$20solution%7Csort:date/or-tools-discuss/bumgTA1QjD0/8Ax8xF0LEQAJ)
We learnt that due to the inherent nature of the problem, the solver was not able to construct even the initial feasible solution to the problem. The suggestion (from the forum) to loosen-up the problem (in terms of constraints) for construction of initial feasible solution was explored and it helped alleviate the problem. Our approach was to allow the solver to construct infeasible solutions in beginning by dropping out few nodes. We implemented this using disjunction constraints. We later penalized for left out nodes thus treating it like a soft instead of hard constraint. Because of this, the solver was not only able to construct the initial feasible solution but also insert back the left out nodes in later iterations. However, as we increased the number of nodes in the range of 150-200 (with proportionate increase in number of vehicles), the solver failed to insert back all the nodes in the solution. This is expected as the solver despite assigning the higher penalty still considers solutions with left out nodes legal. The above observation was with insertion based first solution strategies. Few other first solution strategies, neither found feasible solution nor did claim infeasibility in manageable time (of 30-45 minutes). It is possible that it may have eventually explored the remaining solutions in more time, but we failed to make any such observation.
